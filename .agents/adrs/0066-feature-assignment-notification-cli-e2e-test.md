# 0066. Feature: Assignment Notification CLI E2E Test

## Context

ADR 0065 makes task assignment create an ordinary inbox notification for the
assigned actor. Unit tests cover the action, sync, and CLI surfaces directly,
but the e2e-tests project should also preserve the agent-facing path:

```text
materialize fixture repository
  -> initialize a Situ project from that repository
  -> create an assigned task through the CLI
  -> list notifications for the assigned local agent through the CLI
```

This is the smallest deterministic e2e test for the "agents are asleep until their
inbox has attention" model. It verifies visible records only. It does not run
an external agent CLI, spawn a worker, score model output, or add runtime wake
state.

## Decision

Extend the existing current-repository fixture e2e test to cover
assignment-created notifications. The e2e test should continue to live in the
current repository e2e test rather than creating a separate fixture world.

Expected files:

```text
projects/e2e-tests/src/current-repository-e2e.test.ts
projects/e2e-tests/src/index.ts
```

The e2e test still uses:

- `tinyAutoresearchFixture`
- `materializeFixtureRepository`
- `runSituCli` from `@situ/app`
- a temporary root directory
- an explicit temporary SQLite database path

The e2e test runs against real CLI commands. It must not call app
repositories, app actions, primitive repositories, HTTP handlers, or sync
handlers directly.

## E2E Result Shape

Extend `CurrentRepositoryE2eResult` in `projects/e2e-tests/src/index.ts` with:

```ts
readonly assignedTaskCreateResult: SituCliResult;
readonly notificationListTextResult: SituCliResult;
readonly notificationListJsonResult: SituCliResult;
```

Extend `createCurrentRepositoryE2eResult` input with the same fields and
return those values unchanged.

The helper still attaches `tinyAutoresearchFixture` as `fixture`. It does not
parse CLI output or duplicate product logic.

## Existing E2E Setup

The e2e test creates this deterministic current-repository world before the new
assignment-notification assertions:

```text
databasePath = <rootPath>/situ.db
repositoryPath = materialized.repositoryPath
projectId = project_e2e_current_repository
eventId = event_e2e_current_repository
taskId = task_e2e_current_repository
taskEventId = event_e2e_current_repository_task
actorKind = local_agent
actorId = e2e-agent
now = 2026-05-14T12:00:00.000Z
```

It materializes `tinyAutoresearchFixture` with
`materializeFixtureRepository`, initializes one project from `repositoryPath`,
creates one unassigned task for that project, and verifies:

- `projects current --status active`
- `tasks list --project-id <projectId>`
- `tasks current --project-status active`

The new assigned task is created after the existing `tasks current` text and
JSON assertions so those assertions continue to describe exactly one task.

## New E2E Flow

The e2e test keeps the current-repository flow from ADR 0064. After the existing
`tasks current` text and JSON assertions, add an assigned task and list the
assigned actor inbox.

Use these additional deterministic values:

```text
assignedTaskId = task_e2e_current_repository_assigned
assignedTaskEventId = event_e2e_current_repository_assigned_task
assignedTaskTitle = Review fixture repository
assignedToKind = local_agent
assignedToId = verifier-agent
assignedToDisplayName = Verifier Agent
```

Run assigned task creation:

```text
situ --db <databasePath> tasks create
  --id <assignedTaskId>
  --event-id <assignedTaskEventId>
  --project-id <projectId>
  --title <assignedTaskTitle>
  --body <fixture.goal>
  --actor-kind <actorKind>
  --actor-id <actorId>
  --assigned-to-kind <assignedToKind>
  --assigned-to-id <assignedToId>
  --assigned-to-display-name <assignedToDisplayName>
  --now <now>
```

with `cwd` set to `repositoryPath`.

Run notification list as text:

```text
situ --db <databasePath> notifications list
  --recipient-id <assignedToId>
```

with `cwd` set to `repositoryPath`.

Run notification list as JSON:

```text
situ --json --db <databasePath> notifications list
  --recipient-id <assignedToId>
```

with `cwd` set to `repositoryPath`.

The notification id is generated by the task action and is not deterministic.
The e2e test should assert the id shape and use the observed id when checking
text and JSON output.

## Assertions

The e2e test asserts:

- assigned task creation exits `0`
- assigned task creation text stdout is exactly:

  ```ts
  "Created task task_e2e_current_repository_assigned (event event_e2e_current_repository_assigned_task)\n";
  ```

- notification text listing exits `0`
- notification text listing stdout is exactly one notification line plus a
  trailing newline and no extra blank lines
- the single line has exactly these tab-separated fields:
  - generated notification id, beginning with `notification_`
  - `verifier-agent`
  - `task/task_e2e_current_repository_assigned`
  - `unread`
  - `Assigned task: Review fixture repository`
- notification JSON listing exits `0`
- notification JSON parses as:

  ```json
  {"notifications":[<notification>]}
  ```

- the parsed JSON object has exactly the top-level key `notifications`
- the notifications array contains exactly one notification
- the notification object has no unexpected enumerable keys beyond `id`,
  `recipient`, `target`, `createdBy`, `summaryMarkdown`, and `metadata`
- that notification has:
  - `id`: the same generated id observed in the text output
  - `recipient.recipientId`: `verifier-agent`
  - `recipient.displayName`: `Verifier Agent`
  - `target.targetKind`: `task`
  - `target.targetId`: `task_e2e_current_repository_assigned`
  - `createdBy.actorKind`: `local_agent`
  - `createdBy.actorId`: `e2e-agent`
  - `summaryMarkdown`: `Assigned task: Review fixture repository`
  - no `bodyMarkdown` property
  - no `readAt` property
  - no `dismissedAt` property
  - `metadata.createdAt`: `2026-05-14T12:00:00.000Z`
  - `metadata.updatedAt`: `2026-05-14T12:00:00.000Z`
- `createCurrentRepositoryE2eResult` preserves the assigned task create,
  notification text list, and notification JSON list CLI results

Each non-empty expected stdout includes a trailing newline.

## Boundaries

Do not add app features in this ADR.

Do not add new CLI commands.

Do not change the existing unassigned task assertions described in this ADR.
The new assigned task should be created after those assertions so the earlier
current-task expectations remain a stable one-task check.

Do not call app repositories, app actions, primitive repositories, HTTP
handlers, or sync handlers directly from this e2e test.

The e2e test may call `runSituCli` in-process. It must not invoke external
shell or process commands such as `git`, package managers, benchmark commands,
local agent CLIs, provider SDKs, or network calls.

Do not add LLM evals, LLM judges, model scoring, scheduler behavior, workers,
leases, provider sessions, subagent orchestration, or hidden workflow state.

Do not mark the notification read or dismiss it. This e2e test only proves that
the assigned actor can see the new inbox attention record.

Do not write outside the test temporary root except for ordinary tooling output
such as coverage artifacts.

The gate commands below are developer and CI checks. The e2e test itself must not
invoke them.

## Gates

The e2e test must run as part of:

```text
mise run e2e-tests
```

The root gates must continue to pass:

```text
mise run check
mise run coverage
git diff --check
```

## Consequences

The e2e test now preserves the minimal local-agent wake path:

```text
task assigned to verifier-agent
  -> visible notification exists
  -> verifier-agent can list its inbox through situ notifications list
```

This strengthens the end-to-end regression suite without adding workflow
orchestration or hidden runtime state.
