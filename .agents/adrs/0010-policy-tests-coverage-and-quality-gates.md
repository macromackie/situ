# 0010. Policy: Tests, Coverage, and Quality Gates

## Context

Situ should be safe for agents to change in small slices. The project does not
need a blanket 100% coverage rule, but it does need a reliable way to see test
coverage across the whole repo and confidence that core flows are covered.

Quality gates should be easy to run locally and in CI.

## Decision

Use Bun's test runner for deterministic unit, integration, and e2e tests.

Tests are for deterministic correctness. They should be the default tool for
package contracts, CLI behavior, HTTP/sync behavior, database behavior, and
coverage. LLM evals are a separate quality mechanism defined by ADR 0011.

Root tests run with:

```text
mise run test
```

Deterministic e2e tests run with:

```text
mise run e2e-tests
```

Coverage runs with:

```text
mise run coverage
```

The full local gate runs with:

```text
mise run check
```

`mise run check` must include tests. Coverage is a separate command because it
is slower and produces reports.

## Coverage Contract

Coverage output should include:

```text
coverage/
  lcov.info
  summary.json
```

`summary.json` should include line, function, and branch coverage sections for
the repo. Each section should make clear whether the metric was measured and
whether its threshold was enforced. It may include per-file detail when the tool
supports it.

The coverage summary may be generated by a small repository script over
`lcov.info` when the test runner does not emit the exact JSON shape directly.

Generated files, `node_modules`, `dist`, test files, fixtures, and tiny package
marker files may be excluded from coverage totals.

## Coverage Bar

Coverage does not need to be 100%.

Target thresholds:

- lines: 80%
- functions: 80%
- branches: 70%

`mise run coverage` enforces the line and function thresholds when those totals
are present, and fails when line or function totals are absent. It enforces the
branch threshold only when branch totals are present. When branch totals are not
measured by the coverage provider, the branch section remains present in
`summary.json` with enforcement marked false.

## Core Flow Coverage

Core flows must have tests before they are considered implemented:

- creating and reading projects
- creating, claiming, updating, and completing tasks
- comments and handoff notes
- notifications as an inbox
- experiments and revisions
- measurements, artifacts, and reviews
- report generation from durable records
- CLI commands over app actions
- HTTP/sync surfaces when they exist
- stale-work maintenance when it exists

Package tests should cover package-local contracts. App tests should cover
cross-package flows through app actions.

CLI flows that need a realistic repository, temporary database, or multiple
commands should live in `projects/e2e-tests/` when they become too broad for an
app-local CLI test.

Do not put deterministic CLI regression checks in `projects/evals/`. Evals are
reserved for real LLM tests with LLM judges.

## Consequences

Every package with source code should have at least one test file, even if the
early test only protects the scaffold contract.

Later feature ADRs should add tests with the feature implementation. If a
feature cannot be meaningfully tested yet, the ADR or implementation must state
what blocks useful coverage.
